# ASST4: File systems

Make SFS crash-consistent: journal metadata updates and recover the
volume at mount time, so a machine that dies mid-workload comes back
with a filesystem that is consistent — without scanning the whole
disk. Getting there means first building the filesystem layers this
tree ships without.

Start from: an accepted ASST3.

## Provenance

This restates Harvard CS161's ASST4 (file system journaling and
recovery). Harvard handed out substantial infrastructure — the extra
filesystem syscalls, SFS directory support, fine-grained locking, a
buffer cache, and a ready-made on-disk journal container — and the
graded work was the record schema, the journal instrumentation of
SFS, and recovery. That handout is not public, so here the
prerequisites are part of the exercise. The base tree plans for
exactly this: `os161/design/assignments.txt` describes "basic file
system functionality" and "file system recovery via journaling" as
the two closing assignments, and the tree already ships the crash-
testing tools (frack, poisondisk, sfsck, sys161's doom counter) that
exist for this work. Expect the largest exercise of the sequence.

## Part 1: filesystem syscalls

`sync`, `fsync`, `fstat`, `ftruncate`, `remove`, `rename`, `link`,
`mkdir`, `rmdir`, `getdirentry` — semantics in `os161/man/syscall/`,
wired through the ASST2 file table. Mostly thin wrappers over
`VOP_*`/`vfs_*` calls. emufs already supports directories, so all of
these are testable on the hosted volume before SFS can do any of it.

## Part 2: SFS directories

SFS is currently flat: `mkdir`/`rmdir`/`getdirentry` are wired to
fail-stubs (see `sfs_dirops` in `kern/fs/sfs/sfs_vnops.c`) and rename
never has to cross a directory. Implement real directory trees:
mkdir/rmdir, getdirentry, `.` and `..` in lookup, and cross-directory
rename (the lock-ordering problem that makes rename the hardest vnode
op). Keep `mksfs`, `dumpsfs`, and `sfsck` (`userland/sbin/`) in step
with anything you change on disk — sfsck must walk and repair
directory trees.

## Part 3: fine-grained locking

The VFS layer serializes on `vfs_biglock` (`kern/vfs/vfslist.c`).
Retire it in favor of real locking: per-vnode locks, a vnode-table
lock, a freemap lock, and a stated ordering discipline for the
hierarchical cases (parent/child, `..` lookup, cross-directory
rename). `dirconc` on an SFS volume must show real concurrency, not
serialization.

## Part 4: buffer cache

Put a write-back block cache under SFS: all data and metadata I/O
goes through it. Bounded memory; eviction; `sync` flushes everything;
`fsync` flushes one file; no deadlock between buffer locks and
filesystem locks; operations on distinct blocks proceed concurrently.
Success makes things worse before it makes them better: dirty blocks
now reach disk long after, and in a different order than, the
operations that produced them — that is the crash-consistency problem
Part 5 fixes.

## Part 5: journaling and recovery

Write-ahead journaling for metadata updates, and recovery that
processes the journal at mount time.

- **On-disk journal**: reserve a region of the volume for a log of
  records. The format change is yours to design; update
  mksfs/dumpsfs/sfsck to match.
- **Write-ahead invariant**: the record describing a metadata change
  must be on disk before the block containing the change is.
- **Records**: physical/record-style journaling is fine. Design the
  schema yourself; checkpoint and trim so the log neither fills up
  nor overwrites records recovery still needs.
- **Recovery**: at mount, before anything else touches the volume,
  read the journal and redo/undo to a consistent state.

Ground rules, from the original assignment:

- Some I/O may be made synchronous where justified; all of it may
  not.
- A crash may lose recently written user data — except data covered
  by a completed `sync`/`fsync`, which must be durable.
- The filesystem structure must never be corrupt after recovery, and
  uninitialized block contents must never become visible to userland.
- Recovery works from the journal; it may not scan the whole volume.
- A crash during recovery must leave the volume recoverable.

A design document is required before implementation (in `notes/`,
dated): journal record schema, write-ordering rules, checkpoint
policy, recovery algorithm, locking plan, and how recovery will be
verified. Harvard weighted the design at 30% of the grade; treat it
as a deliverable, not a formality.

## Test tools (all in-tree)

- **Doom counter**: `sys161 -D N kernel` stops the machine dead at
  the Nth disk write. The swap disk is exempt (`nodoom` on slot 2 in
  `root/sys161.conf`), so N counts filesystem writes only. Sweep N to
  crash a workload at every interesting point.
- **frack** (`testbin/`): paired workload/verify tool. `frack do
  <workload>` runs a named workload in the current directory; after a
  crash and recovery, `frack check <workload>` verifies that what is
  on the volume is consistent with a crash at some legal point.
  `frack list` names ~45 workloads: sequential and random writes,
  truncations, holes, file/dir/tree creation and removal, links,
  same- and cross-directory renames, and mixed sequences.
- **poisondisk** (`testbin/`, host-buildable): fills a disk image
  with 0xa9 bytes. Poison the image *before* `mksfs`; any 0xa9 that
  later shows up in user-visible data is uninitialized-block
  exposure.
- **sfsck** (`sbin/`): must report a clean volume after every
  unmount and after every recovery.

## Acceptance

Staged, like ASST3. SFS volumes live on the second disk (`lhd1:`,
`LHD1.img`); make one with `mksfs`, mount with `mount sfs lhd1:` at
the kernel menu. Stages 2–4 run on 1 and 4 CPUs.

1. **Syscalls** (on emufs): the new syscalls work from userland;
   `badcall` passes on them; `dirtest`, `dirseek`, `rmdirtest`,
   `rmtest` pass against the hosted volume.
2. **SFS directories + locking**: the same directory tests plus
   `filetest`, `f_test`, `bigfile`, `sparsefile`, and `dirconc`, all
   on an SFS volume; unmount; `sfsck` reports the volume clean.
3. **Buffer cache**: stage 2 again, plus: `sync` and `fsync` make
   data durable (verifiable with the doom counter — crash after a
   completed sync and the synced data survives); `khu` shows no
   leaks; earlier stages' tests still pass.
4. **Crash recovery**: for a representative spread of frack workloads
   (writes, truncates, holes, tree create/remove, links, renames —
   include the cross-directory ones): poison the image, `mksfs`,
   `frack do` under `sys161 -D N` for a sweep of N values; remount
   (recovery runs), then `frack check` passes and `sfsck` is clean —
   every time, with no poison exposed. Crash *during* recovery (doom
   again with small N), recover again, still clean. Two SFS volumes
   mounted and exercised at once. All previous stages still pass.
