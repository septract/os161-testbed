# ASST3: Virtual memory

Replace dumbvm with a real VM system: physical memory management, TLB
fault handling, paging to disk, and `sbrk`. When this works, the
`GENERIC` kernel config becomes usable and `malloc` works in userland.

Start from: an accepted ASST2.

## What dumbvm does (and why it must die)

`kern/arch/mips/vm/dumbvm.c`: allocates physically-contiguous memory
per segment, never frees anything, maps everything with a fixed
stack of 12 pages, panics on any fault it can't map. `ram.c` hands out
physical pages and can never take them back. The interface to replace
is `kern/include/vm.h` + `kern/include/addrspace.h`; the kernel config
already has hooks (`conf.kern`, the `dumbvm` option) for swapping in
your implementation.

## To implement

1. **Coremap**: a physical-page table tracking every page of RAM
   (owner address space / kernel, virtual address, state). Replaces
   `ram_stealmem`. Must support multi-page kernel allocations
   (`alloc_kpages(n)`) and, later, eviction. Boots from
   `ram_getsize`.
2. **Address spaces + TLB**: per-process page tables (design choice —
   justify it); `vm_fault` handles TLB misses by installing entries
   from the page table; TLB must be handled correctly across context
   switches (shootdown/invalidation or ASIDs). Regions: code, data,
   heap, stack, each growable where appropriate, with proper
   permissions.
3. **sbrk**: the heap syscall, so userland `malloc` works
   (`userland/lib/libc/stdlib/malloc.c` shows what it expects).
4. **Paging**: when RAM runs out, evict: swap disk (`lhd0`), a swap
   map, page-out/page-in paths, a replacement policy better than
   "panic". Pages must be locked against concurrent eviction while in
   flight; no deadlocks between page locks and address-space
   operations.

## Acceptance

Three stages, matching how the course versions grade it:

1. **Coremap only** (no swap yet): `km1`–`km4` pass; `khu` reports no
   leaks after tests; kernel boots and all earlier tests still pass.
2. **Address spaces + sbrk** (generous RAM, e.g. `ramsize=16M`):
   `p /testbin/sbrktest`, `malloctest`, `huge`, `sort`, `matmult`,
   `ctest`, `triplemat`, `palin`; `forktest` and `badcall` still pass.
3. **Swapping** (tight RAM, e.g. `ramsize=1M`/`2M`): the same programs
   plus `triplehuge`, `triplesort`, `parallelvm`, `bigfork` complete —
   slowly is fine, crashing is not.

All on 1 and 4 CPUs. Build the `GENERIC` kernel config as the final
check — that is the "you no longer need dumbvm" milestone.
